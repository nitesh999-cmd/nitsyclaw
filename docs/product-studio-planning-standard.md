# Product Studio Planning Standard

Use this standard before building a new product, service, major revenue feature, landing page, pitch, or AppSumo-style offer. It is for strategy-first work where the idea must be validated before code changes.

Do not use this standard for small bug fixes, reliability patches, known backlog items, or already-scoped implementation tickets.

## Full Version

Act as a world-class product studio, startup strategy team, competitive intelligence analyst, UX architect, software architect, growth marketer, revenue operator, investor pitch deck creator, and harsh devil's advocate.

Use specialist agents to research, critique, improve, and stress-test every part of the idea before building.

Goal:
Turn this product/service idea into a validated, revenue-focused, build-ready execution plan for Codex, Claude Code, or an equivalent AI coding agent.

The output must clearly explain:
- What we are building
- Who it is for
- Why it matters
- Whether competition exists
- Who the best competitors are
- How we are better than the best
- What gaps exist in the market
- How we become 10x better first
- How we become 100x better later
- How we make money
- How much we can realistically make
- How we build it
- How we launch it
- How we explain it in a premium PPTX deck

Do not start coding until the strategy, MVP, competitor gaps, revenue model, and build plan are clear.

### Phase 1 - Define The Product

Answer:
- What exactly are we building?
- What problem does it solve?
- Who is the target customer?
- Why would they care now?
- Why would they pay for it?
- What is the simplest MVP version?
- What should we avoid building early?
- What does success look like in the first 30, 90, and 180 days?

### Phase 2 - Market And Competitor Research

Use research agents to identify:
- Direct competitors
- Indirect competitors
- Best-in-class products
- Pricing models
- Features
- Positioning
- Customer complaints
- Weaknesses
- Missing features
- Bad UX patterns
- Opportunities competitors are ignoring

Create a competitor table with:
- Competitor
- Target customer
- Price
- Core offer
- Main features
- Strengths
- Weaknesses
- Customer pain points
- Opportunity for us
- How we beat them

### Phase 3 - Gap Analysis

Find the highest-value market gaps.

Rank each gap by:
- Customer pain
- Willingness to pay
- Speed to build
- Ease of execution
- Difficulty for competitors to copy
- Defensibility
- Revenue potential
- Marketing angle
- Speed to market

Then recommend:
- The best gap to attack first
- The easiest wedge into the market
- The fastest path to paid users
- The highest-leverage feature
- The biggest mistake to avoid

### Phase 4 - 10x / 100x Differentiation

Explain:
- How we become 10x better first
- What would make us 100x better later
- What our unfair advantage is
- Why customers would switch to us
- Why competitors may struggle to copy us
- What proof or trust signals we need
- What features should be built now
- What features should be delayed
- What features should never be built

### Phase 5 - Revenue Model

Compare:
- Subscription
- One-time payment
- Lead generation
- Commission
- Marketplace
- Usage-based pricing
- Enterprise licensing
- Done-for-you service model
- Hybrid model

Recommend the best revenue model.

Create revenue scenarios:
- Conservative
- Likely
- Aggressive

For each scenario include:
- Number of users/customers/leads
- Conversion rate
- Price per user/customer/lead
- Monthly revenue
- Annual revenue
- Gross margin assumption
- Main assumptions
- Biggest risk
- What must be true for this scenario to happen

### Phase 6 - MVP Specification

Create a tight MVP spec with:
- Core user journey
- Must-have features
- Nice-to-have features
- Features to delay
- Features to avoid
- Admin features
- Database needs
- Integrations
- Security/compliance needs
- Analytics/events to track
- Success metrics

Rule: if it does not help validation, revenue, trust, or conversion, delay it.

### Phase 7 - Build Plan For Coding Agents

Create a build plan that Codex, Claude Code, or an equivalent AI coding agent can execute.

Include:
- Recommended tech stack
- Folder/file structure
- Database schema
- API routes
- Pages/screens
- Components
- Auth requirements
- Payment requirements
- Email/SMS requirements
- Analytics setup
- Testing plan
- Deployment plan

Break work into small tickets.

Each ticket must include:
- Ticket title
- Goal
- Files likely affected
- Exact implementation notes
- Acceptance criteria
- Test steps
- Risk level
- Dependencies
- Definition of done

### Phase 8 - Go-To-Market Plan

Create:
- First 7 days plan
- First 30 days plan
- First 90 days plan
- First 6 months plan
- Launch channels
- AppSumo strategy
- Product Hunt strategy
- Cold outreach strategy
- Paid ads test strategy
- Partnership strategy
- SEO/content strategy
- Social media strategy
- Landing page angle
- First offer
- Pricing test
- First 100 users strategy
- First 10 paying customers strategy

### Phase 9 - Devil's Advocate Review

Use a harsh critique agent to attack the idea.

Cover:
- Why this may fail
- What assumptions are weak
- Where competitors beat us
- Where customers may not care
- Where the product may become too complex
- Where competitors can copy us
- Legal/compliance risks
- Technical risks
- Pricing risks
- Distribution risks
- Trust risks
- What must be proven early

Then improve the plan based on the critique.

### Phase 10 - Final Strategic Output

Produce:

1. One-page executive summary
2. Product definition
3. Customer profile
4. Competitor gap report
5. Differentiation strategy
6. MVP build spec
7. Revenue model
8. Financial scenarios
9. 90-day execution roadmap
10. 6-month roadmap

Slides:
1. Title

4. Target customer
5. Why now
6. Market opportunity
7. Competitor landscape
8. Gaps we found
9. Why we win
10. Product demo flow
11. MVP roadmap
12. Build timeline
13. Go-to-market plan
14. Revenue model
15. Financial scenarios
16. Risks and mitigation
17. 30/60/90-day timeline
18. 6-month roadmap
19. Strengths and unfair advantages

Deck style:
- Premium
- Sharp
- Clean
- Investor-ready
- Simple language
- Strong visuals
- Clear tables

- Clear numbers
- No fluff
- No generic startup language

### Phase 12 - Coding Agent Execution Rules

Before editing code:
- Inspect the current repository.
- Read README, package.json, env examples, database schema, and existing routes.
- Identify the current architecture.
- Create a short implementation plan.
- Do not rewrite working code unnecessarily.

During implementation:
- Make the smallest safe changes.
- Preserve existing conventions.
- Add clear error handling.
- Add analytics where useful.
- Add tests or manual test steps.
- Avoid fake data unless clearly marked.
- Avoid over-engineering.
- Prioritise speed to validation and revenue.

Before final response:
- Run lint/build/tests where available.
- Summarise files changed.
- Explain what was implemented.
- Explain what was not implemented and why.
- List any assumptions made.
- List what still needs human input.
- List recommended next tasks as numbered options.

Final output format:
1. Strategy
2. Competitor gaps
3. Revenue model
4. MVP spec
5. Build tickets
6. Go-to-market plan
7. Risk review
8. PPTX deck outline
9. Next actions

Important rules:
- Do not give generic advice.
- Make assumptions explicit.
- Use real competitor evidence where possible.
- Prefer speed to revenue over feature bloat.
- Prefer validated MVP over perfect product.
- Every recommendation must explain why.
- Every build task must include acceptance criteria.
- Every claim about competitors should be backed by evidence where possible.
- End with the next 5 highest-leverage actions.
